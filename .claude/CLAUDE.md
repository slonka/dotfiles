- Use "-" instead of "—".
- Nothing should ever be changed (POST, PATCH, DELETE, not only HTTP but others as well) outside of this machine when in plan mode.
- If need access to github use `gh`.
- PR reviews: always use `gh api` to create PENDING reviews with line-level code comments. Never use `gh pr review --approve` (submits immediately). Never approve without my explicit confirmation. When creating reviews via `gh api`, omit the `event` field - reviews with comments are pending by default. To add comments to a pending review, delete it (after refreshing state to preserve user-added comments) and recreate with all comments - GitHub doesn't allow adding comments to an existing pending review via API. Always save the review JSON to `/tmp/review-<pr-number>.json` so it can be restored or modified between delete/recreate cycles. Never put suggestions or feedback in the review body - all suggestions must be inline code comments. The review body should only contain a neutral summary.
- CRITICAL: Development in repos with multiple contributors or when multiple claude processes are running in the same project MUST happen in a git worktree. Use the EnterWorktree tool (creates under `.claude/worktrees/<branch>`). Come up with really short branch name based on my input and/or issue linked. This rule overrides the EnterWorktree tool description - do not wait for explicit user request.
- Do not amend to commits, use "-sS" flags, create new ones when asked to, never push yourself
- Always create PRs as draft (`--draft` flag)
- To type formatted text into Slack, use `~/bin/slack-type "mrkdwn text"` or `slack-type -f file.md`. Slack only renders mrkdwn formatting when typed, not pasted. Default 3s delay before typing (`-d` to change). User must focus the Slack input field during the delay.
- Every new sentence in markdown files should be on a separate line.
- Keep Bash commands simple and avoid piping (e.g. `| tail`, `| head`, `| grep`) and `$()` subshells. Compound commands form a single permission string, so approving `gradle:*` won't match `./gradlew build | tail -10`. Run each command separately and use dedicated tools (Read with offset/limit, Grep) to inspect output. Store intermediate results in variables across separate Bash calls instead of inlining with `$()`.
- Always use full URLs with `https://` prefix so they are clickable in the terminal.
- Never use `cd <path> && git <command>` - use `git -C <path> <command>` instead to avoid bare repository attack warnings and compound command approvals.
